## Investigation Phase (2026-02-09)

### Bug Report
- User: Prestataire receives PERMISSION_DENIED when sending messages
- Error: "You do not have permission to send messages in this conversation"
- Additional: Title displays "Discussion generale" instead of "Gestionnaire"

### Root Cause Analysis

**Bug 1: PERMISSION_DENIED (3 layers)**
1. Application: `conversation-service.ts:checkThreadAccess()` - No fallback for `provider_to_managers`
2. Application: `ensureInterventionConversationThreads()` - Provider NOT added as participant to individual thread
3. Database: `can_view_conversation()` RLS - Only checks participants + managers, not assigned providers

**Bug 2: Title Display**
- `getConversationDisplayInfo()` logic is CORRECT (returns "Gestionnaire" for provider)
- Root cause: `getThreadAction` uses user's Supabase client → RLS blocks → thread is null → title fallback
- Fix: Adding provider as participant (US-003) fixes RLS access → thread loads → correct title

**Bug 3: DB Trigger LIMIT 1 (discovered during implementation)**
- `add_assignment_to_conversation_participants` uses LIMIT 1 for individual threads
- With multiple providers, it picks the wrong thread (should filter by participant_id)

### Key Learnings
1. **Three layers of access control**: App (checkThreadAccess) + App (participants) + DB (RLS)
2. **participant_id ≠ participant**: Thread's `participant_id` column identifies the thread owner, but doesn't grant access. Must also be in `conversation_participants` table.
3. **LIMIT 1 vs participant_id filter**: Old trigger assumed one thread per type, broke with individual threads (2026-02-01 migration).
4. **intervention-service.ts was already correct**: `assignUser()` and `assignMultipleProviders()` already add participants. Bug was only in `ensureInterventionConversationThreads` and `create-manager-intervention`.

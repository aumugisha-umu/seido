# Umumentum AI Starter Kit — Design Document

> **For Claude:** REQUIRED SUB-SKILL: Use sp-executing-plans to implement this plan task-by-task.

**Goal:** Create a GitHub repo `umumentum-ai-starter-kit` that serves as a full-stack project scaffolder powered by Claude Code — generates both the project structure (per official stack docs) AND an optimized Claude Code harness. Supports both new projects (full scaffold + landing page + auth) and existing projects (detect + optimize `.claude/`).
**Architecture:** A `.claude/` template with an onboarding skill that interviews the user, scaffolds the project via official CLIs, and generates stack-specific Claude config. For existing projects, detects current setup and enhances it.
**Tech Stack:** Node.js scripts, Markdown, JSON configs. No external dependencies.

## The Onboarding Flow

### Trigger

CLAUDE.md detects first run via a marker file (`.claude/.initialized`). If absent, it instructs Claude to invoke `sp-onboarding`.

```markdown
# CLAUDE.md

> If `.claude/.initialized` does NOT exist, invoke `sp-onboarding` BEFORE any other action.
> After onboarding completes, all sections below are populated.

## Boundaries
[Generated by onboarding]

## Project Structure
[Generated by onboarding]

...
```

### Phase 1 — Interview (sp-onboarding asks, user answers)

Questions asked one at a time:

```
Q1: Project name + one-line description
    → Used for: CLAUDE.md header, projectbrief.md, package.json/Cargo.toml/etc.

Q2: Primary stack (multiple choice + custom)
    → Next.js 15 / React + Vite / Python FastAPI / Ruby on Rails /
      Go + Chi / Rust + Axum / SvelteKit / Nuxt.js / Other: ___
    → Used for: scaffold command, directory structure, rules

Q3: Database (multiple choice)
    → Supabase (Postgres + RLS) / PostgreSQL raw / MongoDB /
      SQLite / MySQL / None
    → Used for: DB rules, migration commands, types generation

Q4: Key libraries/frameworks (free text, comma-separated)
    → Examples: shadcn/ui, Tailwind, Stripe, Prisma, Drizzle, etc.
    → Used for: context7 lookups, rules generation

Q5: Testing framework preference (auto-suggested based on stack)
    → Vitest / Jest / Pytest / RSpec / Go test / Playwright E2E / Cypress
    → Used for: tester agent examples, commands section

Q6: Deployment target (multiple choice)
    → Vercel / AWS / Railway / Fly.io / Docker / Self-hosted / Other
    → Used for: build commands, CI config awareness

Q7: Additional context (open-ended, optional)
    → Team size, special constraints, existing code to integrate
```

### Phase 2 — Scaffold (Claude executes autonomously)

#### Step 2a: Official CLI scaffold

Based on Q2 answer, run the official project scaffolder:

| Stack | Command |
|-------|---------|
| Next.js 15 | `npx create-next-app@latest . --typescript --tailwind --eslint --app --src-dir --import-alias "@/*"` |
| React + Vite | `npm create vite@latest . -- --template react-ts` |
| Python FastAPI | `mkdir -p app/{routers,models,schemas,services,core} tests && touch app/__init__.py app/main.py requirements.txt` |
| Ruby on Rails | `rails new . --database=postgresql --css=tailwind --skip-jbuilder` |
| Go + Chi | `go mod init {name} && mkdir -p cmd/server internal/{handlers,models,services,middleware} pkg` |
| SvelteKit | `npx sv create . --template minimal --types ts` |
| Nuxt.js | `npx nuxi init .` |

If project files already exist (non-empty directory), SKIP scaffolding and enter **Existing Project Mode** (see below).

#### Step 2b: Database setup

| DB | Command |
|----|---------|
| Supabase | `npx supabase init` → creates `supabase/` dir with config + migrations |
| Prisma | `npx prisma init` → creates `prisma/schema.prisma` |
| Drizzle | `mkdir drizzle && touch drizzle.config.ts` |
| Raw Postgres | `mkdir migrations` |

#### Step 2c: Context7 documentation fetch

For each library from Q2+Q3+Q4:
1. `resolve-library-id` → get context7 ID
2. `query-docs` → fetch key patterns, project structure recommendations
3. Use findings to generate accurate rules and examples

#### Step 2d: Generate Claude Code harness

Using interview answers + context7 findings, generate all files:

**CLAUDE.md** — Populated with:
- Boundaries (3-tier, adapted to stack)
- Project Structure (from actual scaffolded dirs)
- Commit Workflow (git* trigger, LIGHT/FULL)
- Development Commands (from stack's CLI)
- Development Rules (from stack conventions)

**Rules files** (path-scoped):
- `rules/code-quality.md` — universal (reuse search, anti-patterns)
- `rules/testing-rules.md` — stack-specific test patterns
- `rules/db-rules.md` — only if DB selected, with stack-specific ORM patterns
- `rules/api-rules.md` — only if API framework, with route patterns

**Agent files** — With Example Output sections using the actual stack:
- `agents/frontend-developer.md` — only if frontend stack
- `agents/backend-developer.md` — with stack-specific server patterns
- `agents/database-analyzer.md` — only if DB selected
- `agents/tester.md` — with actual test framework examples

**Memory bank** — Pre-filled:
- `projectbrief.md` — from Q1 + Q7
- `techContext.md` — full stack description, commands, conventions
- `activeContext.md` — "Project initialized, ready for first feature"
- `systemPatterns.md` — architecture patterns from context7 docs

**Settings**:
- `settings.json` — MCP servers (context7 always, plus stack-relevant ones)
- `settings.local.json` — hooks wiring, base permissions

#### Step 2e: Landing Page + Auth Frontend (New Projects Only)

For NEW projects with a frontend stack, generate production-ready starter pages following best practices from UI design, UX research, copywriting, and SEO:

**Landing Page** (`app/page.tsx` or equivalent):
- Hero section with value proposition headline (copywriting best practices)
- Features section with 3-4 key benefits
- CTA section with clear call-to-action
- Footer with links
- Mobile-first responsive design
- SEO meta tags (title, description, Open Graph, Twitter Card)
- Semantic HTML (accessibility-first: headings hierarchy, landmarks, alt text)
- Stack-specific styling (Tailwind if available, CSS modules otherwise)

**Auth Pages** (if auth library detected or selected):
- Login page with email/password + social OAuth buttons
- Signup page with form validation
- Forgot password page
- Clean, minimal design following the stack's conventions
- Client-side validation with appropriate library (Zod, Yup, etc.)

**Design Principles Applied:**
- Anti-AI-slop: No generic gradients, no "Welcome to [App]" headlines, distinctive visual identity
- UX Research: Clear information hierarchy, minimal cognitive load, obvious next actions
- Copywriting: Benefit-driven headlines, specific CTAs, no filler text
- SEO: Proper heading structure, meta tags, semantic HTML, fast loading
- Mobile-first: Touch-friendly tap targets, readable font sizes, no horizontal scroll

**Templates per stack:**

| Stack | Landing | Auth |
|-------|---------|------|
| Next.js 15 | `app/page.tsx` + `app/layout.tsx` with metadata | `app/auth/login/page.tsx`, `app/auth/signup/page.tsx` |
| React + Vite | `src/pages/Home.tsx` + `src/pages/Login.tsx` | React Router routes |
| SvelteKit | `src/routes/+page.svelte` | `src/routes/auth/` |
| Nuxt.js | `pages/index.vue` | `pages/auth/login.vue` |
| Python/Ruby/Go | Skip (API-only stacks) | Skip |

#### Step 2f: Install dependencies + first commit

```bash
# Install deps (if package.json exists)
npm install  # or pip install -r requirements.txt, bundle install, etc.

# Create .initialized marker
touch .claude/.initialized

# First commit
git add .
git commit -m "feat: initial project scaffold with Claude Code harness"
```

## Existing Project Mode

When the target directory already contains files, the onboarding enters **Existing Project Mode** instead of scaffolding from scratch.

### Detection Flow

```
Directory not empty?
  → Check for .claude/ directory
    → EXISTS: Enter "Optimize Mode"
    → MISSING: Enter "Retrofit Mode"
```

### Optimize Mode (`.claude/` exists)

1. **Audit current setup** — Read CLAUDE.md, rules/, agents/, skills/, hooks, settings
2. **Score against best practices** — Check for:
   - Three-tier boundaries (Auto-Apply / Ask First / Never Do)
   - Path-scoped rules with YAML frontmatter
   - Agent Example Output sections (GitHub 2500+ repos #1 factor)
   - Security hooks (block-dangerous-commands, block-secret-writes)
   - Memory bank files (activeContext, techContext, systemPatterns, projectbrief, progress)
   - MCP servers (at minimum context7)
   - Compound learnings setup (LEARNINGS.md or AGENTS.md)
3. **Present optimization report** — Show what's missing/suboptimal
4. **Ask user for approval** before each change
5. **Apply improvements** — Add missing hooks, upgrade CLAUDE.md structure, add agents with Example Output, etc.

### Retrofit Mode (no `.claude/`)

1. **Analyze the project** — Detect stack from package.json, Cargo.toml, requirements.txt, Gemfile, go.mod, etc.
2. **Detect existing patterns** — Read code to identify:
   - Architecture patterns (MVC, service layer, repository, etc.)
   - Testing framework in use
   - Database/ORM setup
   - Auth library
   - Coding conventions (naming, file structure)
3. **Run interview** — Same Q1-Q7 but pre-fill answers from detection (user confirms/corrects)
4. **Generate `.claude/` harness** — Same as new project Phase 2d, but adapted to existing structure
5. **Do NOT scaffold** — No CLI commands, no file creation outside `.claude/`
6. **Do NOT modify existing code** — Only add the `.claude/` harness

## Generated CLAUDE.md Template

The onboarding generates a CLAUDE.md following this structure (< 200 lines):

```markdown

- {Stack-specific patterns: e.g., "Repository Pattern" or "Service layer" or "MVC"}

- {Testing: e.g., "Update test files when modifying routes"}

### Ask First (before implementation)
- New DB migration or schema change
- Adding new external dependency
- Modifying auth/security logic
- Cross-cutting changes (3+ domains)

### Never Do
- git add/commit/push without user typing git*
- npm run build / cargo build without explicit ask
```

## Design System (generated at onboarding)

Onboarding generates a complete design system to prevent component duplication:

| File | Purpose |
|------|---------|
| `docs/design/00-design-system.md` | Component hierarchy, tokens, typography, spacing |
| `docs/design/anti-patterns.md` | UI mistakes to avoid |
| `.claude/rules/design-system-rules.md` | Enforced during development (path-scoped) |
| `.claude/rules/design-evaluation-criteria.md` | Anti-AI-slop scoring rubric |

**Component Hierarchy (5 levels):**
1. **Primitives** (atoms) — UI library components (Button, Input, Badge)
2. **Composites** (molecules) — 2-3 primitives combined (SearchInput, FormField)
3. **Features** (organisms) — Domain-aware (ProjectCard, InvoiceTable)
4. **Templates** (layouts) — Page-level structure (DashboardLayout)
5. **Pages** — Route-level composition + data fetching

Skills NOT included (too SEIDO-specific): sp-ralph, sp-prd, sp-analytics, sp-monitoring, sp-release, all SEO strategy skills.

## Hooks (identical pattern, stack-agnostic)

| Hook | Script | Trigger |
|------|--------|---------|
| SessionStart | `session-startup.js` | Every session — shows context, warnings |
| PreToolUse:Bash | `block-dangerous-commands.js` | Blocks force-push, rm -rf, hard reset |
| PreToolUse:Write | `block-secret-writes.js` | Blocks writes to .env, .pem, .key |
| PostToolUse:Edit\|Write | `track-changes.js` | Queues modified files |
| PostToolUse:Bash | `post-commit-memory-sync.js` | Reminds to sync after commit |
| Stop | `update-active-context.js` | Updates activeContext on session end |

All hooks use async `process.stdin` pattern (Windows-compatible) and `exit(2)` for blocks.

# Progress Log — Fix Subscription Limit False Positives
# Started: 2026-02-26
# Bugs found: 6 | Fixed: 4 critical
---

## Root Cause Analysis
User had 155 subscribed lots, blocked at lot #134. Two combined causes:
1. Off-by-one: client `>=` blocks AT limit, server `<=` allows AT limit
2. Stale `billable_properties`: cached count in subscriptions table could drift from reality

## 2026-02-26 — BUG-001: Off-by-one in isAtLotLimit (>= → >)
- Changed `>=` to `>` in all 3 forms to align with server-side `<=`
- Server: `(actual + count) <= subscribed` = allowed → Client must: `(actual + batch) > subscribed` = blocked
- Files: lot-creation-form.tsx, building-creation-form.tsx, edit-building-client.tsx

## 2026-02-26 — BUG-002: Stale billable_properties cache
- `getLotCount()` read cached `billable_properties` from subscriptions table
- Trigger only fires on INSERT/UPDATE OF deleted_at/DELETE — misses team_id changes, bulk ops
- FIX: Changed to live `COUNT(*)` on lots table with `head: true` (exact count, no data transfer)
- Files: lib/services/repositories/subscription.repository.ts

## 2026-02-26 — BUG-003: duplicateLot bypasses limit check
- `duplicateLot()` in lot-creation-form.tsx had NO `isAtLotLimit()` check
- Added check + upgrade modal trigger
- Files: lot-creation-form.tsx

## 2026-02-26 — BUG-004: addIndependentLot bypasses limit check
- `addIndependentLot()` had NO limit check at all
- Also: `isAtLotLimit()` didn't count `independentLots.length` — only counted `lots.length`
- FIX: Added check to addIndependentLot + combined both arrays in isAtLotLimit
- Files: lot-creation-form.tsx

## Learnings
- Three different limit operators across codebase: `<` (checkCanAddProperty private), `<=` (canAddProperty public), `>=` (client). Must be consistent
- Cached counters (billable_properties) drift — prefer live counts for critical business logic
- Duplicate/clone functions often bypass validations added to the "add" function
- Independent lots mode was completely unprotected by subscription limits

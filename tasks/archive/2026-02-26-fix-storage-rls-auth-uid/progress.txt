# Progress Log — Fix Storage RLS auth.uid() vs users.id Mismatch
# Started: 2026-02-26
# Stories: 4/4 complete
---

## Root Cause Analysis
The `documents` and `email-attachments` bucket RLS policies (migrations 20260220150000 and 20260225100000) use:
```sql
WHERE tm.user_id = auth.uid()
```
But `team_members.user_id` stores `users.id` (database PK), NOT `auth.uid()` (Supabase Auth UUID).
These are DIFFERENT UUIDs. The legacy `intervention-documents` bucket correctly uses `get_current_user_id()`.
All roles (admin, gestionnaire, locataire, prestataire, proprietaire) exist in `team_members.role`.
The service role bypass was a WRONG workaround — the real fix is to use `get_my_profile_ids()` in the RLS.

## 2026-02-26 — US-001+002: Fix storage RLS policies
- Created: 20260226110000_fix_storage_rls_auth_uid_vs_users_id.sql
- Replaced `tm.user_id = auth.uid()` with `tm.user_id IN (SELECT get_my_profile_ids())` in 6 policies
- Affected buckets: `documents` (INSERT, SELECT, DELETE) + `email-attachments` (INSERT, SELECT, DELETE)
- Kept `documents_update` unchanged — uses `owner_id = auth.uid()::text` (correct for storage.objects)
- Added `owner_id = auth.uid()::text` to DELETE policies (both buckets) for error-recovery cleanup
- Files: supabase/migrations/20260226110000_fix_storage_rls_auth_uid_vs_users_id.sql

## 2026-02-26 — US-003: Revert service role bypasses
- Reverted create-intervention/route.ts: `serviceClient` → `supabase` for fileService.uploadInterventionDocument()
- Reverted upload-intervention-document/route.ts: removed `storageClient`, upload+cleanup use `supabase`
- Reverted conversations/upload/route.ts: removed `storageClient`, upload+cleanup+signedUrl use `supabase`
- Removed `createServiceRoleSupabaseClient` import from 2 routes (kept in create-intervention for thread creation)
- Files: create-intervention/route.ts, upload-intervention-document/route.ts, conversations/upload/route.ts

## 2026-02-26 — US-004: MEMORY.md corrections
- Fixed "Two role systems": team_members.role has ALL 5 roles, not just gestionnaire/admin
- Added new learning: "Storage RLS: auth.uid() vs users.id"
- Updated E2E entry: service role no longer needed for storage uploads

## Learnings
- **auth.uid() ≠ users.id**: Supabase Auth UUID is in `users.auth_user_id`, database PK is `users.id`. `team_members.user_id` stores `users.id`. Storage RLS must use `get_my_profile_ids()` to bridge.
- **get_my_profile_ids() > get_current_user_id() for RLS**: Multi-profile users need IN (set), not = (single). `get_current_user_id()` picks ONE arbitrarily.
- **storage.objects.owner_id IS auth.uid()**: No conversion needed for owner checks. `owner_id = auth.uid()::text` is correct.
- **Legacy bucket got it right**: `intervention-documents` policies use `get_current_user_id()`. New `documents` policies used `auth.uid()` directly — regression.
- **Service role bypass masks RLS bugs**: Using service role hides the real issue. Fix RLS first, then verify if bypass is truly needed.
- **team_member_role enum**: admin, gestionnaire, locataire, prestataire, proprietaire — ALL roles exist in team_members

# Progress Log — Fix Subscription Limit False Positives
# Started: 2026-02-26
# Bugs found: 6 | Fixed: 4 critical
---

## Root Cause Analysis
User had 155 subscribed lots, blocked at lot #134. Two combined causes:
1. Off-by-one: client `>=` blocks AT limit, server `<=` allows AT limit
2. Stale `billable_properties`: cached count in subscriptions table could drift from reality

## 2026-02-26 — BUG-001: Off-by-one in isAtLotLimit (>= → >)
- Changed `>=` to `>` in all 3 forms to align with server-side `<=`
- Server: `(actual + count) <= subscribed` = allowed → Client must: `(actual + batch) > subscribed` = blocked
- Files: lot-creation-form.tsx, building-creation-form.tsx, edit-building-client.tsx

## 2026-02-26 — BUG-002: Stale billable_properties cache
- `getLotCount()` read cached `billable_properties` from subscriptions table
- Trigger only fires on INSERT/UPDATE OF deleted_at/DELETE — misses team_id changes, bulk ops
- FIX: Changed to live `COUNT(*)` on lots table with `head: true` (exact count, no data transfer)
- Files: lib/services/repositories/subscription.repository.ts

## 2026-02-26 — BUG-003: duplicateLot bypasses limit check
- `duplicateLot()` in lot-creation-form.tsx had NO `isAtLotLimit()` check
- Added check + upgrade modal trigger
- Files: lot-creation-form.tsx

## 2026-02-26 — BUG-004: addIndependentLot bypasses limit check
- `addIndependentLot()` had NO limit check at all
- Also: `isAtLotLimit()` didn't count `independentLots.length` — only counted `lots.length`
- FIX: Added check to addIndependentLot + combined both arrays in isAtLotLimit
- Files: lot-creation-form.tsx

## Learnings
- Three different limit operators across codebase: `<` (checkCanAddProperty private), `<=` (canAddProperty public), `>=` (client). Must be consistent
- Cached counters (billable_properties) drift — prefer live counts for critical business logic
- Duplicate/clone functions often bypass validations added to the "add" function
- Independent lots mode was completely unprotected by subscription limits
